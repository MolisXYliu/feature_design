## 一、主流技术路线对比

| 技术路线 | 原理 | 路由延迟 | 成本节约 | 冷启动 | 实现复杂度 | 适合阶段 |
|---------|------|---------|---------|--------|-----------|---------|
| **规则路由** | if/else 条件分支、关键词/模式匹配 | ~0ms | 30-50% | 即用 | 低 | MVP |
| **复杂度路由** | 评估 prompt 复杂度，简单任务走便宜模型 | 5-20ms | 45-85% | 需标注数据 | 中 | 中期 |
| **分类器路由** | 小型 ML 模型（BERT/MF/LLM）做路由分类 | 5-100ms | 35-85% | 需训练数据 | 中-高 | 中期 |
| **语义路由** | Embedding 相似度匹配到预定义路由路径 | 5-20ms | ~48% | 需示例集 | 中 | 中期 |
| **级联/降级路由** | 从便宜模型开始尝试，质量不达标则逐级升级 | N/A（多次调用） | 最高 98% | 即用 | 中 | 成熟期 |
| **成本感知路由** | 满足质量约束下选成本最低的模型 | ~0ms | 30-70% | 需成本数据 | 低-中 | 全程（横切） |
| **强化学习路由** | 将路由建模为序列决策，RL 优化策略 | 5-20ms | 研究中 | 需大量数据 | 高 | 成熟期 |

**行业趋势**：实际生产系统几乎都采用**混合策略**——规则路由做 fast path + 分类器处理模糊样本 + 成本感知做横切约束 + fallback 保可靠性。没有任何一家只用单一路线。

## 二、业界产品实践

### 2.1 同类产品（代码/Agent 场景）

| 产品 | 路由技术 | 核心机制 | 关键特点 |
|------|---------|---------|---------|
| **GitHub Copilot** | 启发式 + 基础设施信号 | 问一系列规则问题：是快速补全？需多步推理？需宽上下文？然后匹配模型能力级别 | 当前主要按模型可用性选择，task-aware 能力标注为"coming soon"；模型过重/受限时自动降级 |
| **Cursor IDE** | 启发式 + 质量检测 | Auto 模式检测到输出质量下降 → 逐消息自动切换模型 | 明确声明非 task-aware；部分请求（如摘要）强制路由到 OpenAI；使用 Turbopuffer 向量引擎做上下文检索（非路由） |

### 2.2 技术标杆

| 产品 | 路由技术 | 核心机制 | 可借鉴点 |
|------|---------|---------|---------|
| **Notion AI** | 任务类别 + fine-tuned 专用模型 | 按任务类型（写作/搜索/自动填充）固定路由；高频低复杂度任务训练专用小模型 | **Braintrust trace 驱动评估闭环**——AI 团队 80% 工作基于 trace；fine-tuned 专用模型使延迟减半、质量提升 |
| **OpenAI GPT-5** | Mixture of Models（内部路由） | 统一系统"知道何时快速回答、何时深度思考"；用户手动切模型行为作为路由偏好训练信号 | **Mixture of Models** 是行业公认的下一代范式；从用户行为持续学习路由策略 |

### 2.3 路由平台

| 产品 | 路由技术 | 核心机制 | 关键特点 |
|------|---------|---------|---------|
| **OpenRouter** | 黑盒 Auto Router + Provider 负载均衡 | meta-model 分析 prompt 后路由到候选模型；Provider 按价格倒数平方加权随机化 | 500+ 模型池；零加价透明定价；`:nitro`（最高吞吐）/`:floor`（最低价格）路由变体 |
| **Martian** | 机制可解释性（SAE + Circuit Analysis） | 将 LLM 黑盒转换为可解释形式，分析模型内部结构预测其在特定 query 上的表现 | 无需实际运行目标模型即可预测表现；上层提供 Embedding 路由降本 40-70% |
| **Unify.ai** | 神经网络 + 实时 benchmark | 每 10 分钟更新 benchmark（含延迟/成本/地域/并发维度）；神经网络在 benchmark 上训练质量预测模型 | 用户可设 quality/cost/latency 三维权重；支持上传私有数据训练专属路由器 |
| **Not Diamond** | Embedding + 随机森林（RoRF） | prompt → embedding → Random Forest 预测每个模型胜率 → 选概率最高的模型 | 概率性软路由（非硬二分）；60ms 路由延迟；RoRF 开源；支持自定义训练 |

### 2.4 云厂商托管路由

| 平台 | 路由能力 | 说明 |
|------|---------|------|
| **Amazon Bedrock** | 同模型家族内按 prompt 路由 | 声称不明显损失准确率下降本最高 30% |
| **Azure AI Foundry** | 统一端点多模型智能选择 | 企业治理/合规友好 |
| **Google Vertex AI** | `RoutingConfig` 支持 balanced/quality/cost 偏好 | 与 Vertex AI 生态深度集成 |

> 三大云厂商均已在 2025 年将模型路由作为**平台级能力**提供，说明模型路由正从"高级优化"变成"基础设施标配"。

### 2.5 行业关键洞察

| # | 洞察 |
|---|------|
| 1 | 没有统一的"标准路由器"——每家公司根据自身约束采用完全不同的架构 |
| 2 | 共同模式：**规则/启发式做 fast path + 质量评估做反馈闭环** |
| 3 | **Embedding + 分类器**是最主流的可复制技术路线（Martian / Not Diamond / RouteLLM） |
| 4 | **路由即服务（RaaS）** 正在成为独立赛道——OpenRouter $5 亿估值 |
| 5 | Fine-tuned 专用模型是降本利器（Notion 实践验证），但需足够任务量支撑 |

## 三、开源框架对比

| 框架 | 语言 | 路由方式 | 核心体积 | 延迟开销 | Provider 支持 | Fallback | Node.js 集成 | 适合场景 |
|------|------|---------|---------|---------|-------------|----------|-------------|---------|
| **LiteLLM** | Python | 6 种负载均衡 + Embedding auto-routing | 300-400MB | 20-40ms | 100+ | 完善（可组合） | 需 Python 子进程 | Python 后端 |
| **Portkey AI Gateway** | TypeScript | 条件路由 + 加权负载均衡 | 122KB | <1ms | 200+ | 完善（可组合） | **原生 npm 包** | Node.js/TS 项目 |
| **RouteLLM (LMSYS)** | Python | 5 种 ML 分类器（MF/BERT/LLM/SW/Random） | 轻量 | 5-100ms | 2（强/弱对） | 无 | 需 Python | 研究/评估 |
| **Semantic Router** | Python | Embedding 相似度匹配 | 轻量 | 5-20ms | N/A（路由层） | 无 | 需 Python | 嵌入应用 |
| **Not Diamond RoRF** | Python | Embedding + 随机森林分类器 | 轻量 | ~60ms | 12 组预训练对 | 无 | 需 Python | 研究/参考 |
| **NVIDIA LLM Router Blueprint** | Python | 意图路由（Qwen 1.75B）+ CLIP embedding + 神经网络 | 较重（含模型） | 50-200ms | 可配置 | 有 | 需 Python | 企业级/GPU 环境 |
| **Ruflo** | TypeScript | 三层递进：WASM 引擎（正则意图匹配 + AST 代码变换）→ 四维特征加权评分选模型 → 关键词匹配升级强模型 | 中等 | <1ms ~ 2s | Claude 系列 | 有（电路断路器） | **原生 TS** | Agent 协调平台 |

**框架选型结论**：

| 评估维度 | 结论 |
|---------|------|
| 直接可用 | 无框架完全匹配（Electron + Node.js + LangGraph + 桌面分发场景） |
| 设计参考 | Portkey（条件路由 + 可组合 fallback）、RouteLLM（ML 分类器思路）、RoRF（embedding + 分类器） |
| 核心约束 | 桌面应用排除 Python 重依赖；需深度集成 LangGraph checkpoint；需处理副作用安全 |
| 推荐路径 | **自建路由内核**，借鉴开源框架设计理念 |

## 四、落地方案建议

### 4.1 推荐架构

三层路由架构：**硬约束过滤 → 规则路由 → 语义打分**，逐层递进，高置信快速返回。

```
用户请求 → 模式判断（固定/自动）
                    ↓ 自动模式
           Layer 1: 硬约束过滤 (<1ms)
           API key 有效性 / 模型健康 / 上下文窗口 / 工具支持
                    ↓ 排除不可用模型
           Layer 2: 规则路由 (<1ms)
           任务类型 / 消息长度 / 代码检测 / 工具关键词
                    ↓ 低置信时
           Layer 3: 语义打分 (10-30ms，Phase 3+)
           本地 embedding + 线性打分
                    ↓
           resolvedModelId → 锁定到本次 Run，resume 不重新路由
```

**设计原则**：

| 原则 | 说明 |
|------|------|
| 调用前路由 | 兼容现有单模型 runtime，最小改动 |
| 保留手动选择 | 固定模型/自动路由双模式，不替换用户习惯 |
| Run 级冻结 | 一次执行中模型不变，避免副作用场景的行为漂移 |
| 后台任务优先 | Heartbeat/定时任务/Memory Summarize 是最快见效的优化点 |
| 可观测先行 | 没有数据就没有优化依据 |

### 4.2 分阶段路线

| Phase | 内容 | 周期 | 核心产出 | 预期收益 |
|-------|------|------|---------|---------|
| **Phase 0** | 数据基建与模型元数据 | 1 周 | 模型 tier/成本/能力元数据；调用追踪管道；路由配置结构 | 为后续优化奠定数据基础 |
| **Phase 1** | 规则路由上线 | 1-2 周 | `resolveExecutionPlan()` 路由入口；后台任务默认降级；UI Auto 模式 | **后台任务成本降 40-60%** |
| **Phase 2** | Fallback + 成本监控 | 1 周 | 429/超时自动降级；成本监控面板 | 提升可用性；成本可视化 |
| **Phase 3** | 语义打分器 | 2 周 | 本地 embedding + 线性打分（仅低置信时触发）；隐式信号采集 | 覆盖规则无法判断的模糊场景 |
| **Phase 4** | 工具裁剪 + 路由缓存 | 1-2 周 | 按任务类型裁剪工具注入；路由决策缓存 | 额外降低 10-15% token 成本 |
| **Phase 5** | 数据驱动优化 | 持续 | 效果回流闭环；阈值自校准；模型自动 profiling | **综合 TCO 降低 30-50%** |

### 4.3 路由效果回流与持续优化闭环

路由系统的价值不仅在初始规则的准确率，更在于**能从真实使用数据中持续自我优化**。以下是完整的效果回流机制设计：

#### 4.3.1 在线信号采集

| 信号类型 | 采集方式 | 信号强度 | 用途 |
|---------|---------|---------|------|
| 用户直接采纳输出 | 对话行为追踪 | 强正面 | 验证路由选择正确 |
| 用户手动编辑输出 | 编辑距离计算 | 弱负面 | 模型可用但非最优 |
| 用户手动切换模型 | UI 事件（override） | **强负面** | 路由选错了，最直接的纠正信号 |
| 用户重试/重新发送 | 对话行为 | 强负面 | 输出质量不达标 |
| 用户停止生成 | 对话行为 | 强负面 | 方向完全错误 |
| 工具调用成功率 | Agent 执行日志 | 客观指标 | Agent 场景核心能力验证 |
| JSON/Schema 合规率 | 输出格式校验 | 客观指标 | 结构化输出能力验证 |
| 首 token 延迟 / 总延迟 | 系统指标 | 客观指标 | 用户体感评估 |
| Token 用量 / 实际成本 | 计费统计 | 客观指标 | TCO 追踪 |

#### 4.3.2 离线评估机制

| 机制 | 原理 | 触发时机 | 产出 |
|------|------|---------|------|
| **Trace 回放** | 用历史 prompt 回放新路由策略，对比新旧策略的成本和质量差异 | 路由规则变更前 | 预期成本变化 / 质量影响评估 |
| **Shadow Routing** | 新策略与旧策略并行打分，但只执行旧策略的结果 | 新策略验证期 | 新旧策略差异率、override 预测 |
| **LLM-as-Judge 抽样** | 定期用强模型对路由结果做质量评估（采样，非全量） | 周期性（如每周） | 各 tier 模型在不同任务类型上的质量评分 |
| **模型自动 Profiling** | 新模型上线时跑一组固定基准 prompt，记录延迟/质量/工具成功率 | 模型上下线时 | 模型能力画像（capabilities / quality prior / latency prior） |

#### 4.3.3 闭环优化路径

```
┌──────────────────────────────────────────────────────────────┐
│                    持续优化闭环                               │
│                                                              │
│   在线信号 ──┐                                               │
│              ├→ 信号聚合 → 规则阈值调优 ──→ Trace 回放验证    │
│   离线评估 ──┘            │                      │           │
│                           │               Shadow Routing     │
│                           │                      │           │
│                           ↓                 Canary 发布      │
│                    语义权重训练                   │           │
│                    (Phase 5)              全量上线            │
│                           │                      │           │
│                           └──────────────────────┘           │
│                                                              │
│   模型上下线 → 自动 Profiling → 更新模型能力画像 → 调整路由   │
└──────────────────────────────────────────────────────────────┘
```

**具体优化动作**：

| 回流数据 | 优化动作 | 示例 |
|---------|---------|------|
| 用户 override 率高（某任务类型） | 调整该任务类型的路由规则/提升模型 tier | "代码重构"任务 override 率 40% → 将其默认路由从 standard 提升到 premium |
| 某模型工具调用成功率低 | 降低该模型在工具密集型任务上的优先级 | 模型 X 工具调用成功率仅 65% → 有工具场景自动避开 |
| 某模型延迟波动大 | 动态调整 health 权重 | 模型 Y 的 P99 延迟从 3s 飙到 15s → 临时降权 |
| 隐式信号积累足够 | 训练语义权重（embedding scorer） | 积累 1000+ 条带标签数据 → 训练 semanticWeights，方案 C 进化为方案 B |
| 新模型上线 | 自动 Profiling → 分配 tier 和 capabilities | 新增模型 Z → 跑 50 条基准 prompt → 自动标注 tier=standard, capabilities=[tools, json] |

#### 4.3.4 发布安全机制

```
规则/阈值变更 → Trace 回放（离线）→ Shadow Routing（在线不执行）→ Canary 5%（在线执行）→ 全量
```

每个阶段监控的关键指标：

| 指标 | 回滚阈值 |
|------|---------|
| 用户 override 率 | > 当前基线 + 10% |
| 工具调用成功率 | < 当前基线 - 5% |
| Fallback 触发率 | > 当前基线 + 15% |
| 平均延迟 | > 当前基线 + 30% |
